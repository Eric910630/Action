## 信用点使用说明

### 不同操作的信用点消耗

根据操作类型，每次操作消耗的信用点不同：

- **Scrape（单页抓取）**：5-10 个信用点（取决于方案）
- **Extract（结构化提取）**：通常比 Scrape 消耗更多信用点（因为涉及 LLM 处理）
- **Crawl（网站爬取）**：按页面数量计算
- **Batch Scrape（批量抓取）**：按 URL 数量计算

### 信用点计算示例

**免费方案**：
- 每次 Scrape：5 个信用点
- 500 credits ÷ 5 = 100 次 Scrape/月

**爱好者方案**：
- 每次 Scrape：10 个信用点
- 3,000 credits ÷ 10 = 300 次 Scrape/月

**标准方案**：
- 每次 Scrape：50 个信用点
- 100,000 credits ÷ 50 = 2,000 次 Scrape/月

## Action 项目使用建议

### 当前配置

根据 Action 项目的使用场景：

1. **热点增强功能**：
   - 默认只增强前 10 个热点
   - 使用 `extract_hotspot_details` 方法
   - 每次提取消耗约 10-20 个信用点（取决于页面复杂度）

2. **月度使用量估算**：
   - 假设每天抓取 1 次热点，每次增强 10 个
   - 每月约 30 次 × 10 个 = 300 次提取
   - 每次 15 个信用点（平均值） = 4,500 credits/月

## 成本优化建议

### 1. 选择性增强

当前实现已经优化：
- ✅ 只增强前 10 个热点（避免过多 API 调用）
- ✅ 使用并发控制（`max_concurrent=3`）
- ✅ 错误处理（失败时保留原始数据）

### 2. 缓存利用

Firecrawl 支持缓存（`maxAge` 参数）：
- 可以设置缓存时间，减少重复请求
- 相同 URL 的重复请求会使用缓存，不消耗信用点

### 3. 监控使用量

在 Firecrawl Dashboard 中：
- 实时查看信用点使用情况
- 设置使用量警告阈值
- 监控 API 调用频率

### 4. 环境变量配置

可以设置信用点警告阈值：

```bash
# 在 .env 文件中（如果使用 MCP）
FIRECRAWL_CREDIT_WARNING_THRESHOLD=2000    # 警告阈值：2000 credits
FIRECRAWL_CREDIT_CRITICAL_THRESHOLD=500    # 严重警告：500 credits
```

## 注意事项

1. **定价可能变化**：以上定价信息仅供参考，实际价格可能因时间、地区等因素有所变化，建议访问 [Firecrawl 官网](https://www.firecrawl.dev/pricing) 获取最新信息。

2. **信用点过期**：免费方案的信用点通常按月重置，付费方案的信用点可能按订阅周期重置。

3. **超出限制**：如果超出信用点限制，API 调用会失败，需要升级方案或等待下个周期重置。

4. **自托管选项**：Firecrawl 是开源的（AGPL-3.0 许可证），如果需要大规模使用，可以考虑自托管部署。

## 自托管 vs Cloud

### Cloud（云服务）

- ✅ 无需维护
- ✅ 自动更新
- ✅ 高可用性
- ❌ 需要付费（超出免费额度）
- ❌ 有速率限制

### 自托管（Self-hosted）

- ✅ 完全免费（无信用点限制）
- ✅ 完全控制
- ✅ 无速率限制（取决于服务器性能）
- ❌ 需要维护
- ❌ 需要服务器资源

**建议**：
- 小规模使用：Cloud（免费或爱好者方案）
- 大规模使用：考虑自托管或企业方案
